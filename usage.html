구성하기 — 작업을 어디에 두나

흩어진 대화 대신, 한 주제를 한곳에.

Projects — 한 주제를 한 공간에

언제
한 가지 일(예: "블로그 운영", "이사 준비")의 대화·자료·규칙을 매번 새로 설명하기 싫을 때.
예시
Projects에서 새 프로젝트 → 관련 파일을 지식(Knowledge Base)에 업로드 → 프로젝트 지침에 "어떻게 답해줘"를 적기. 그 프로젝트 안의 모든 대화가 이 자료·지침을 공유.
함정
⚠
무료는 프로젝트 최대 5개. 자료를 한 프로젝트에 너무 많이 넣으면 답이 산만해질 수 있어요 — 주제별로 나누세요.

전역 지침 vs 프로젝트 지침

언제
늘 지킬 규칙(말투·언어 등)과 이 프로젝트만의 규칙을 구분하고 싶을 때.
예시
계정 전역 "Instructions for Claude" = 모든 대화에 적용("항상 한국어로, 간결하게"). 프로젝트 지침 = 그 프로젝트 대화에만 적용("이 자료 기준으로만 답해").
함정
⚠
예전 "스타일(Styles)" 기능은 Skills로 이관 중입니다. 새로 시작한다면 스타일 대신 스킬로 잡으세요.

실행하기 — 무엇으로 시키나

구성했으면, 이제 능력을 붙여 실제로.

Skills 사용 — 반복 작업을 능력으로

언제
매번 똑같이 시키는 작업(문서 만들기, 표 정리 등)을 버튼 하나처럼 쓰고 싶을 때.
예시
Settings → Capabilities에서 Code execution & file creation 켜기 → Customize → Skills에서 원하는 스킬 토글. 이후 대화에서 자동 호출되거나 직접 지정.
함정
⚠
코드 실행 토글이 꺼져 있으면 작동 안 함. 코드 실행·스킬은 Free 포함 전 플랜 가능(토글만 켜면 됨, 유료 아님).

Connectors(MCP) 사용 — 내 도구 불러오기

언제
캘린더·노션·드라이브 같은 외부 도구를 대화 안에서 직접 다루고 싶을 때.
예시
설치편 L2에서 커넥터를 추가했다면, 채팅 입력창 "+"에서 그 커넥터를 켜고 "다음 주 일정 잡아줘"처럼 사용.
함정
⚠
무료는 커스텀 커넥터 1개. 신뢰할 수 있는 서버만 연결(외부 콘텐츠 = prompt injection 위험).

Research — 여러 출처를 묶어 깊게 조사

언제
한 번의 검색으로 안 되는, 여러 출처를 교차해야 하는 조사가 필요할 때.
예시
채팅 "+" → Research 켜기 → 질문하면 웹과 연결된 서비스(Gmail·캘린더 등)를 스스로 여러 번 검색해 인용 달린 답을 줌.
함정
⚠
유료(Pro·Max·Team·Enterprise) 기능이고, 웹 검색이 켜져 있어야 작동. 웹·데스크톱·모바일에서 사용 가능.

구성하기 — 폴더와 지침을 어디에 두나

Claude Code는 "어디서 켰는지"와 "지침을 어디 뒀는지"로 움직입니다.

작업폴더 — 한 폴더 = 한 작업

언제
Claude Code를 켤 때마다 "여기가 무슨 프로젝트인지"를 명확히 해두고 싶을 때.
예시
프로젝트 폴더(보통 git repo) 루트에서 claude 실행. Claude Code는 그 폴더와 상위 폴더의 지침을 자동으로 읽습니다.
함정
⚠
"폴더를 어떻게 나누나"엔 공식 정답이 없습니다(여기 '권장' 배지). 한 작업 = 한 폴더로 시작하고, 커지면 하위 폴더에 그 부분용 CLAUDE.md를 두는 방식이 무난.

전역 vs 프로젝트 지침 — CLAUDE.md 계층

언제
어디서나 지킬 규칙은 전역에, 이 repo만의 규칙은 프로젝트에 두고 싶을 때.
예시
전역(내 모든 프로젝트): ~/.claude/CLAUDE.md
프로젝트(이 repo, 팀 공유): ./CLAUDE.md 또는 ./.claude/CLAUDE.md
개인 메모(공유 안 함): ./CLAUDE.local.md
함정
⚠
이들은 서로 덮어쓰지 않고 전부 합쳐져(누적) 읽힙니다(넓은 것 먼저, 가까운 것 나중). 그러니 전역과 프로젝트가 서로 모순되게 쓰지 마세요.

전역 vs 프로젝트 — 스킬 · MCP 배치

언제
스킬·MCP를 나만 쓸지(전역) vs 이 repo·팀이 쓸지(프로젝트) 정할 때.
예시
스킬: 개인 ~/.claude/skills/ vs 프로젝트 .claude/skills/
MCP: claude mcp add --scope user|project|local … · 팀 공유는 repo 루트 .mcp.json
함정
⚠
프로젝트 스코프로 둔 스킬·MCP는 그 폴더에서 켰을 때만 보입니다. "왜 안 보이지?"의 흔한 원인.

한 번에 다 하지 말 것 — 점진 layering

지침·문서 체계는 "지향점"이지 1일차 과제가 아닙니다. 막힘없이 쌓는 순서:

  • 1일차: 전역 지침 1개(말투·원칙) + 프로젝트 CLAUDE.md 한 장. 끝.
  • 익숙해지면: 프로젝트 용어집 CONTEXT.md — 헷갈리는 단어를 한 곳에 정의.
  • 결정이 쌓이면: docs/adr/ — "왜 이렇게 정했나"를 짧게 기록(ADR).
  • 일이 많아지면: 이슈 트래커(.scratch/)로 할 일을 파일로 관리.
  • 팀·반복이 생기면: triage 라벨로 들어오는 일감을 분류.

실행하기 — 어떤 모드로 시키나

같은 일도 "어떻게 굴리느냐"가 다릅니다. 입문 핵심 3가지부터.

플랜모드(Plan Mode) — 바꾸기 전에 계획부터

언제
큰 변경·위험한 작업 전에, Claude가 곧장 파일을 고치기 전 계획을 먼저 보고 싶을 때.
예시
Shift+Tab으로 모드 순환(default → acceptEdits → plan). plan 모드에선 읽기·탐색만 하고 계획을 제안 → 승인하면 그때 실제 수정 시작.
함정
⚠
승인 전에는 아무것도 안 고칩니다(그게 핵심). 계획이 어긋났으면 Ctrl+G로 직접 고친 뒤 진행.

subagent — 곁가지 일은 따로 시켜 요약만 받기

언제
검색·로그·파일 탐색처럼 중간 산출물이 많은 곁가지가 메인 대화를 어지럽힐 때.
예시
"이 코드베이스에서 X 찾아줘"를 자식 에이전트에게 위임 → 그 에이전트가 자기 컨텍스트에서 처리하고 요약만 메인에 반환(메인 대화는 깔끔).
함정
⚠
한 세션 안에서 도는 도우미입니다 — 결과를 메인에 보고할 뿐, 자기들끼리 대화하지 않습니다(그건 agent team).
더
→ 심화편: 커스텀 subagent 만들기·도구 제한·모델 라우팅

deep-research — 출처 교차검증 리포트

언제
여러 출처를 모아 교차검증된 인용 리포트가 필요할 때(단순 검색 이상).
예시
>/deep-research <질문>
Claude Code 내장 워크플로가 여러 각도로 검색→출처 교차검증→인용 리포트를 백그라운드로 생성. (앱에선 같은 일을 "+"→Research가 함.)
함정
⚠
웹 검색 도구가 있어야 하고, 여러 에이전트를 돌려 토큰을 꽤 씁니다. 좁은 질문부터.
