추측·과설계·범위이탈을 줄이는 지침

무엇인가

Karpathy Guidelines는 Claude Code·Cursor 같은 AI 코딩 도구에게 “추측하지 말고, 과하게 만들지 말고, 필요한 곳만 고치고, 검증 가능한 목표까지 확인하라”고 알려주는 짧은 행동 지침 모음입니다.

비전공자 비유

운전을 잘하는 사람이 스포츠카를 몰아도 안전벨트와 차선이 필요하듯, 코딩을 잘하는 AI에게도 멈출 때·물어볼 때·검증할 때를 알려줘야 합니다.

기능

네 가지 행동 원칙

지침의 전부 — 이 4가지를 에이전트에게 상기시킵니다.

1

Think Before Coding

애매하면 혼자 결정하지 말고, 가정과 선택지를 드러내고 먼저 질문하게 합니다.

2

Simplicity First

요청하지 않은 기능, 과한 추상화, 미래 대비용 복잡성을 줄입니다.

3

Surgical Changes

수술하듯 필요한 줄만 고치고, 주변 코드를 마음대로 리팩터링하지 않게 합니다.

4

Goal-Driven Execution

“고쳐줘”를 테스트와 성공 기준이 있는 검증 가능한 작업으로 바꿉니다.

어디에 쓰나

이럴 때 켜두면 좋습니다

  • AI에게 코드 작성을 맡길 때 — 기능 추가·버그 수정·리팩터링 전반에 두루 적용됩니다.
  • AI가 자꾸 과하게 설계할 때 — 단순한 함수를 큰 구조로 만들거나 불필요한 옵션을 붙이는 것을 줄입니다.
  • 변경 범위가 너무 커질 때 — 요청한 것 외에 주석·포맷·주변 코드까지 바꾸는 문제를 막습니다.
  • 버그 수정의 성공 기준이 흐릴 때 — “재현 테스트 → 수정 → 재검증” 흐름으로 바꿉니다.
  • 여러 AI 에이전트를 병렬로 굴릴 때 — 각 에이전트가 같은 작업 원칙을 따르게 하는 공통 규칙으로 좋습니다.

맥락

왜 이게 인기일까

인기 이유는 AI를 갑자기 천재로 만들어서가 아닙니다. 이미 코드를 잘 쓰는 AI가 성급하게 추측하고, 과하게 만들고, 관련 없는 코드까지 바꾸는 습관을 줄여주기 때문입니다.

앞으로 개발자는 모든 코드를 직접 치기보다 여러 에이전트에게 일을 나누고, 좋은 지침과 검증 루프를 설계하는 능력이 중요해집니다. 이 스킬은 그중 가장 기본적인 “에이전트 작업 규칙” 역할을 합니다.

# Karpathy Guidelines

Behavioral guidelines to reduce common LLM coding mistakes, derived from
Andrej Karpathy's observations on LLM coding pitfalls.

**Tradeoff:** These guidelines bias toward caution over speed.
For trivial tasks, use judgment.

## 1. Think Before Coding
**Don't assume. Don't hide confusion. Surface tradeoffs.**
- State your assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them - don't pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- If something is unclear, stop. Name what's confusing. Ask.

## 2. Simplicity First
**Minimum code that solves the problem. Nothing speculative.**
- No features beyond what was asked.
- No abstractions for single-use code.
- No "flexibility" or "configurability" that wasn't requested.
- No error handling for impossible scenarios.
- If you write 200 lines and it could be 50, rewrite it.

Ask: "Would a senior engineer say this is overcomplicated?" If yes, simplify.

## 3. Surgical Changes
**Touch only what you must. Clean up only your own mess.**
- Don't "improve" adjacent code, comments, or formatting.
- Don't refactor things that aren't broken.
- Match existing style, even if you'd do it differently.
- If you notice unrelated dead code, mention it - don't delete it.

The test: Every changed line should trace directly to the user's request.

## 4. Goal-Driven Execution
**Define success criteria. Loop until verified.**
- "Add validation" -> "Write tests for invalid inputs, then make them pass"
- "Fix the bug" -> "Write a test that reproduces it, then make it pass"
- "Refactor X" -> "Ensure tests pass before and after"

Strong success criteria let you loop independently.
Weak criteria ("make it work") require constant clarification.
