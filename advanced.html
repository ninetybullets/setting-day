한 명으론 부족할 때,
여럿을 굴린다.

사용편(개발자 트랙)을 끝냈다면, 여기선 여러 에이전트를 어떻게 굴리는지를 봅니다. 위임(subagent) → 협업(agent team) → 자동화(workflow), 점점 규모가 커지는 3단 계단입니다.

⌘
여기는 Claude Code 전용입니다. 세 기능 모두 터미널·IDE의 Claude Code 기능으로, Claude 챗 앱엔 없습니다. 사용편의 실행 모드(플랜모드·subagent 맛보기)를 이미 써봤다는 전제로 한 단계 더 들어갑니다.
1

subagent — 곁가지를 격리해 위임

토대. 나머지 둘이 여기 올라탑니다.

무엇이고 언제 — 자식 에이전트에게 위임

언제
검색·로그·파일 탐색처럼 중간 산출물(컨텍스트)이 많은 곁가지를 메인 대화에서 떼어내고 싶을 때.
동작
각 subagent는 자기만의 fresh·격리된 컨텍스트 윈도우에서 일하고 요약(결과)만 메인에 돌려줍니다 → 메인 컨텍스트가 깔끔하게 보존됨. (예외: fork는 부모 컨텍스트를 물려받음.)
호출
자동 위임(Claude가 작업을 보고 알아서 띄움) 또는 명시 호출 — at-mention @이름 / --agent 플래그.
함정
⚠
한 세션 안에서 도는 도우미입니다. 메인에 보고만 할 뿐 서로 대화하지 않습니다 — teammate끼리 대화하는 건 2단계 agent team.

커스텀 만들기 — .claude/agents/<이름>.md

구조
Markdown 파일 + 상단 YAML frontmatter. 필수는 name·description 둘뿐, 나머지는 선택. 본문엔 그 에이전트의 시스템 프롬프트를 적습니다.
--- .claude/agents/researcher.md --- name: researcher description: 코드베이스 탐색·검색 전담. 곁가지 조사에 위임. tools: Read, Grep, Glob # 허용 목록(allowlist) model: haiku # 저가 모델로 라우팅 --- # 여기에 이 에이전트용 시스템 프롬프트를 적는다
도구·모델
tools = 허용 목록(allowlist), disallowedTools = 차단 목록(denylist). model로 haiku 같은 저가 모델을 지정해 비용↓(생략 시 부모 모델 상속).
스코프
저장 위치가 범위를 정합니다 — 프로젝트 .claude/agents/(이 repo·팀) vs 유저 전역 ~/.claude/agents/(내 모든 프로젝트).
함정
⚠
description이 자동 위임을 좌우합니다. "언제 이 에이전트를 쓰는지"를 또렷이 적어야 Claude가 제때 띄웁니다. 막연하면 안 불려요.
2

agent team — teammate끼리 직접 협업

subagent가 "보고만" 한다면, 여기선 서로 대화합니다.

여러 세션이 한 팀으로

언제
하나로 안 끝나는 여러 작업 흐름을 장기간 병행하며, 일꾼끼리 서로 조율·분담해야 할 때.
동작
team lead(메인 세션) + teammate(들)가 공유 task list로 일을 나누고, mailbox로 서로 직접 메시지를 주고받습니다(subagent와 결정적 차이).
켜기
기본 꺼짐 — 실험 기능입니다. 환경변수로 활성화:
$export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
함정
⚠
실험적·고비용. 2026-06 기준 여전히 실험 기능(기본 OFF)이고, 동시에 여러 세션을 돌려 토큰을 크게 씁니다 — 플랜 모드 기준 단일 세션의 약 7배. 정말 병행 협업이 필요할 때만.
3

workflow — 스크립트가 결정론적으로 자동화

사람이 턴마다 정하는 대신, 스크립트가 흐름을 쥡니다.

수십~수백을 한 번에

언제
코드베이스 전수 감사, 대규모 마이그레이션처럼 "한 대화로 안 되는" 대량 작업을 재현 가능하게 돌릴 때.
동작
Claude가 JS 스크립트를 작성하고, 런타임이 그 스크립트대로 subagent 수십~수백 개를 백그라운드로 오케스트레이션. subagent(1단계)가 여기서 대량으로 fan-out됩니다.
짜임새
스크립트는 meta 블록(이름·설명) + agent()(한 에이전트) · parallel()(동시 실행) · pipeline()(단계 연결) 같은 빌딩블록으로 흐름을 조립합니다. (정확한 함수 스펙은 공식 문서 참조.)
켜기
v2.1.154+, 모든 유료 플랜(Pro는 /config에서 켜기). 프롬프트에 ultracode 키워드를 포함하면 트리거됩니다.
함정
⚠
"dynamic workflow"는 별개 기능이 아니라 이 workflow와 같은 것을 가리키는 이름입니다. 트리거 키워드는 v2.1.160에서 바뀌었으니(이전 workflow → 이후 ultracode) 버전을 확인하세요. 토큰은 가장 많이 씁니다.
무엇을 언제

셋 중 무엇을 고르나

공식 기준은 하나 — "누가 다음 할 일(plan)을 쥐는가."

곁가지를 격리 컨텍스트에 위임하고 요약만 받고 싶다 →
subagent
여러 흐름을 장기 병행하며 일꾼끼리 서로 대화·조율해야 한다 →
agent team (실험)
수십~수백 단위를 재현 가능하게 자동화하고 싶다 →
workflow

먼저 알고 들어가기

강력한 만큼, 새는 곳도 큽니다

심화 기능 셋은 모두 "여럿을 돌리는" 일이라, 입문 모드보다 비용·변동성이 큽니다. 켜기 전에 셋만 기억하세요.

토큰이 빠르게 샙니다

에이전트를 여럿·대량으로 굴리는 구조라 토큰 소모가 큽니다. 특히 agent team은 약 7배, workflow는 run당 수십~수백 개를 띄웁니다. 좁은 범위부터 시작하세요.

실험·버전에 묶입니다

agent team은 실험 기능(CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1), workflow는 v2.1.154+·트리거 키워드가 v2.1.160에서 변경됨. 버전에 따라 동작이 다를 수 있습니다.

"되던 게 또 될" 보장은 다릅니다

workflow는 스크립트라 재현성이 높지만, subagent·agent team은 매 턴 Claude가 판단하므로 같은 입력이 늘 같은 흐름을 보장하진 않습니다. 결정론이 중요하면 workflow.
